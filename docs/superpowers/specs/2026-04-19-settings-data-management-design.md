# 数据管理页面设计规格

**日期**：2026-04-19  
**作者**：Codex  
**状态**：Draft  
**Spec 类型**：页面体验设计 + 数据边界设计 + 风险收敛  
**目标页面**：`lib/presentation/screens/settings/data_management_screen.dart`  
**入口页面**：`lib/presentation/screens/settings/settings_screen.dart`

---

## 1. 背景

当前设置页里的“数据管理”还只是占位入口：

- 还没有独立页面
- 没有区分“历史记录”“缓存”“AI 配置”“提示词模板”“偏好设置”
- 没有导入导出能力
- 没有细粒度清理能力
- 没有危险操作分层

但从产品形态看，这个页面会逐渐成为用户资产管理中心，因为本应用的数据不是一次性输入，而是会持续累积：

- 各术数历史记录
- 用户问事内容
- 自定义 AI 接口配置
- 提示词模板
- 应用偏好

这意味着“数据管理”不能只是一个“清空数据”的按钮集合，而必须明确回答 4 个问题：

1. 用户当前有哪些本地数据
2. 哪些数据能备份、能迁移
3. 哪些数据能清理、能恢复
4. 哪些操作有风险，风险边界在哪里

---

## 2. 设计目标

数据管理页的核心目标不是“显示所有技术项”，而是让用户安全完成以下任务：

1. 知道自己在本地积累了多少数据
2. 备份和迁移自己的核心资产
3. 清理不需要的历史或缓存
4. 在不误删敏感信息的前提下做重置

页面必须同时满足：

- **可理解**：普通用户看得懂
- **可恢复**：危险操作前有清晰拦截
- **可扩展**：未来可增加导入导出、自动备份、云同步
- **可映射到当前代码**：不脱离现有 Drift + SecureStorage 架构

---

## 3. 当前实现约束

为了避免写出脱离仓库现实的方案，先固定现状。

### 3.1 本地持久化分层

当前仓库有两套本地存储：

- `Drift / SQLite`
- `flutter_secure_storage`

其中：

- `DivinationRecords` 表存放历史记录主数据
- `ProviderConfigs` 表存放 LLM 提供者非敏感配置
- `PromptTemplates` 表存放提示词模板
- `UserPreferences` 表存放用户偏好
- `SecureStorage` 存放 API Key 以及若干加密字段

### 3.2 历史记录结构

历史记录通过 `DivinationRepository` 统一读写，已具备：

- 获取全部记录
- 按系统查询
- 按起卦方式查询
- 按时间范围查询
- 删除单条
- 删除指定系统全部记录
- 删除全部记录

这说明“历史记录管理”具备先做页面层的基础条件。

### 3.3 AI 配置结构

当前 AI 配置已完成多命名配置集能力：

- 用户可保存多套接口配置
- 当前激活配置通过 `ai_active_provider_profile_id` 标识
- 所有配置元信息保存在 `UserPreferences`
- 每个 profile 的 API Key 保存在 `SecureStorage`

因此，数据管理页可以安全提供：

- 导出接口配置元信息
- 单独清空接口配置
- 可选导出敏感信息

### 3.4 提示词模板结构

提示词模板已独立存表，且支持：

- 获取全部模板
- 获取按系统模板
- 保存模板
- 删除模板
- 设置激活模板

所以模板是天然可备份、可恢复、可恢复默认值的数据类型。

## 4. 当前架构中的风险与缺陷

数据管理页设计必须直接对准这些问题，不能回避。

### 4.1 风险一：历史清空与敏感存储边界混淆

当前 `DivinationRepositoryImpl.deleteAllRecords()` 在删完数据库记录后，直接调用：

- `SecureStorage.deleteAll()`

这会带来严重副作用：

- 不仅删掉历史记录关联加密字段
- 还会删掉 AI API Key
- 未来若加入更多 secure item，也会被一并删掉

**结论**：

- “清空全部历史记录”绝不能等价于“清空所有 secure storage”
- 数据管理页必须推动后端改成“按命名空间删除”

### 4.2 风险二：数据类型没有被产品化命名

目前技术上已经存在多类数据，但在产品层没有清晰命名：

- 历史记录
- AI 接口配置
- 提示词模板
- 偏好设置
- 缓存

如果页面不先做分类，用户看到的会是一堆“导出”“清除”“恢复默认”，完全不知道影响范围。

### 4.3 风险三：危险操作缺少分级

“清理缓存”和“清空全部数据”完全不是一类动作。

如果页面把它们放在同一层级，会让危险动作显得过于轻易。

### 4.4 风险四：备份范围没有契约

如果未来直接上“导出备份”，但不先定义导出包内容，会出现以下混乱：

- 不知道 API Key 是否导出
- 不知道模板是否包含内置项
- 不知道偏好是否覆盖导入
- 不知道是否支持跨版本导入

**结论**：

- 先定义备份契约，再做按钮

---

## 5. 页面定位

数据管理页应被定义为：

> 本地数据的查看、备份、迁移、清理与恢复中心

它不是：

- 调试页面
- 技术参数页面
- 单纯“清空缓存”页

因此页面的信息架构必须优先按“用户可理解的数据对象”分组，而不是按底层表名分组。

---

## 6. 页面总体结构

建议采用四段式结构：

```text
┌──────────────────────────────┐
│ 数据管理                     │
│ 存储概览卡片                │
├──────────────────────────────┤
│ 备份与迁移                  │
│ 导出备份 / 导入备份         │
├──────────────────────────────┤
│ 历史记录管理                │
│ 按系统清理 / 按时间清理     │
├──────────────────────────────┤
│ 清理与重置                  │
│ 清理缓存 / 恢复默认 / 重置  │
└──────────────────────────────┘
```

用户阅读顺序应该是：

1. 我现在有哪些数据
2. 我能先备份什么
3. 我能清掉什么
4. 我在做危险动作前要确认什么

---

## 7. 页面模块设计

## 7.1 模块一：存储概览

这个模块放在最顶部，用于降低用户对“数据管理”的不确定感。

### 7.1.1 应显示的内容

- 历史记录总数
- 各术数记录数
- 最近一条记录时间
- AI 接口配置数量
- 自定义提示词数量
- 上次备份时间

第一版可以不显示精准磁盘大小，避免一开始引入额外文件系统统计复杂度。

### 7.1.2 展示形式

建议为一块摘要卡片，不做大而重的 dashboard。

建议文案形态：

- `历史记录 128 条`
- `AI 配置 3 套`
- `自定义模板 6 条`
- `上次备份：未备份`

### 7.1.3 P0 与 P1

- P0：记录数量、配置数量、模板数量、最近备份时间
- P1：磁盘占用、最近清理时间

---

## 7.2 模块二：备份与迁移

这是数据管理页的核心价值模块，优先级高于“清空数据”。

### 7.2.1 P0 必需操作

- `导出完整备份`
- `导入备份`

### 7.2.2 导出时的范围选择

导出页不应只有一个“立即导出”，而应支持勾选项：

- 历史记录
- AI 接口配置
- 提示词模板
- 应用偏好

敏感项单独配置：

- `包含 API Key（高风险）`

默认规则：

- 默认不导出 API Key
- 默认导出非敏感配置元信息

### 7.2.3 导入时的模式选择

导入必须显式选择模式：

- `合并导入`
- `覆盖导入`

定义：

- 合并导入：保留现有数据，导入新数据；同 ID 冲突时以导入包为准
- 覆盖导入：先清理目标范围，再写入导入包

### 7.2.4 导入前校验

导入前必须校验：

- 备份包格式合法
- `manifest.json` 存在
- 版本号可识别
- 所含数据类型明确

如果校验失败：

- 不能部分导入
- 必须中止并给出原因

---

## 7.3 模块三：历史记录管理

这个模块只负责“历史记录”，不能把 AI 配置和模板混进来。

### 7.3.1 应包含的操作

- `按术数清理`
- `按时间清理`
- `清空全部历史记录`
- `导出历史记录`

### 7.3.2 按术数清理

提供以下目标：

- 六爻
- 大六壬
- 梅花易数
- 小六壬

每项都显示当前数量，避免盲删。

文案示例：

- `清理大六壬记录（36 条）`

### 7.3.3 按时间清理

P0 建议提供预设：

- 清理 7 天前记录
- 清理 30 天前记录
- 清理 90 天前记录

P1 再增加：

- 自定义时间区间

### 7.3.4 清空全部历史记录

这是危险操作，但危险级别低于“清空全部应用数据”。

确认弹窗必须明确说明：

- 删除的是历史记录
- 不影响 AI 接口配置
- 不影响提示词模板
- 不影响应用偏好

如果当前实现仍会删除全部 secure storage，则**禁止上线此按钮**，必须先修复仓储实现。

---

## 7.4 模块四：清理与重置

这个模块放在页面底部，使用更强的风险提示。

### 7.4.1 P0 推荐项

- `清理缓存`
- `恢复默认提示词`
- `清空 AI 接口配置`
- `清空全部本地数据`

### 7.4.2 清理缓存

缓存必须与“用户资产”明确区分。

缓存可包括：

- 临时图片
- 临时导出文件
- 模型列表缓存
- 临时渲染缓存

缓存清理不应影响：

- 历史记录
- AI 接口配置
- 提示词模板
- 用户偏好

### 7.4.3 恢复默认提示词

这个动作的语义应是：

- 删除用户自定义修改
- 恢复内置模板状态

不应影响：

- AI 接口配置
- 历史记录
- 其他偏好

### 7.4.4 清空 AI 接口配置

影响范围应明确为：

- 删除所有命名配置元信息
- 删除所有 profile API Key
- 清除当前激活配置 ID

不应影响：

- 历史记录
- 提示词模板
- 其他偏好

### 7.4.5 清空全部本地数据

这是页面最危险操作，应满足：

- 红色危险区
- 二次确认
- 输入确认文案后才能执行

确认文案建议：

- 输入 `全部清空` 后才能继续

执行范围：

- 历史记录
- AI 接口配置
- 提示词模板
- 用户偏好
- 缓存

---

## 8. 页面交互细则

## 8.1 列表层级

推荐从上到下：

1. 存储概览卡
2. 备份与迁移组
3. 历史记录管理组
4. 清理与重置组

### 8.2 每个操作项应包含的信息

每个列表项不应只有标题，至少要有：

- 标题
- 副标题说明影响范围
- 是否危险
- 可选统计信息

例如：

- `清空 AI 接口配置`
- `删除所有已保存的接口参数与 API Key，不影响历史记录`

### 8.3 操作完成反馈

每次操作后至少反馈：

- 成功提示
- 处理数量
- 是否可恢复

例如：

- `已删除 42 条大六壬记录`
- `已恢复默认提示词，共重置 12 条模板`

---

## 9. 备份包格式契约

第一版建议统一为：

- `zip` 容器
- 内部使用 `json`

建议结构：

```text
backup.zip
├─ manifest.json
├─ records.json
├─ ai_profiles.json
├─ prompt_templates.json
├─ preferences.json
└─ secure_payload.json   // 仅勾选敏感信息时存在
```

### 9.1 manifest.json

至少包含：

- 应用名
- 备份格式版本
- 导出时间
- 导出范围
- 数据条数统计

### 9.2 records.json

存放：

- `DivinationResult.toJson()` 的稳定结果

禁止存放：

- 临时 UI 字段
- 页面态字段

### 9.3 ai_profiles.json

默认仅包含：

- profile id
- 名称
- provider id
- baseUrl
- model
- temperature
- maxOutputTokens
- createdAt / updatedAt

默认不包含：

- API Key

### 9.4 secure_payload.json

只有当用户显式勾选“包含敏感信息”时才生成。

并且必须：

- 明确标红警示
- 后续支持加密导出

如果第一版做不到加密导出，则建议：

- 第一版直接不支持导出敏感信息

这比“明文导出 API Key”更安全。

## 10. 技术实现建议

## 10.1 新增应用服务层

不建议把导入导出逻辑直接塞进页面。

建议新增：

- `DataManagementService`
- `BackupExportService`
- `BackupImportService`

职责建议：

- `DataManagementService`：聚合统计、清理、重置
- `BackupExportService`：导出包组装
- `BackupImportService`：校验与导入

## 10.2 删除策略必须命名空间化

当前 secure storage 删除逻辑必须改造：

不能再使用：

- `SecureStorage.deleteAll()`

应改为按前缀或显式 key 集合删除，例如：

- 历史记录相关：`question_*`、`detail_*`、`interpretation_*`
- AI 配置相关：`llm_profile_*_apikey`

### 10.3 数据管理页不直接操作底层表

页面层只调用服务层，不直接访问：

- DAO
- Drift 表
- SecureStorage

否则后续导入导出和删除策略会散落到多处。

---

## 11. 分阶段落地建议

### Phase 1：页面与统计

目标：

- 建立 `DataManagementScreen`
- 做存储概览
- 做静态列表结构
- 接通历史记录数量、AI 配置数量、模板数量

### Phase 2：清理能力

目标：

- 按系统清理历史
- 清空全部历史
- 清空 AI 配置
- 恢复默认提示词
- 清理缓存

前置条件：

- 修复 secure storage 删除边界

### Phase 3：备份导出

目标：

- 导出 JSON ZIP 备份
- 导入备份
- 导入校验与模式选择

### Phase 4：增强能力

目标：

- 自定义时间区间清理
- 自动备份
- 备份浏览
- 云同步前置抽象

---

## 12. 验收标准

数据管理页完成后，至少满足以下标准：

1. 用户能明确看懂本地有哪些类型的数据
2. 历史记录、AI 配置、提示词模板的边界清晰
3. 清理缓存与删除用户资产被明确区分
4. 危险操作全部具备确认步骤
5. “清空历史记录”不会误删 AI API Key
6. 备份格式有稳定契约，导入前会做校验
7. 页面结构支持后续扩展，而不是一次性堆按钮

---

## 13. 结论

数据管理页的最优方向，不是先做“更多删除按钮”，而是先把以下问题定义清楚：

- 数据对象有哪些
- 每种数据如何统计
- 哪些能备份
- 哪些能删除
- 删除后的边界是什么

对当前项目来说，最佳路径是：

1. 先做页面信息架构与统计
2. 修复 secure storage 删除边界
3. 再上清理与备份能力

如果不先收敛边界，后续“清空历史”“导出备份”“重置应用”这三类功能会非常容易互相串伤，最终把数据管理页做成一个高风险入口。
