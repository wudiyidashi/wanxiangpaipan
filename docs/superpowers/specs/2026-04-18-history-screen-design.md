# 历史记录页面设计规格

**日期**：2026-04-18  
**作者**：Codex  
**状态**：Draft  
**Spec 类型**：页面体验设计 + 信息架构收敛  
**目标页面**：`lib/presentation/screens/history/history_list_screen.dart`

---

## 1. 背景

当前历史页已经具备基础能力：

- 加载全部记录
- 按术数系统筛选
- 列表展示
- 点击进入详情
- 长按删除
- `chromeless` 模式，可嵌入首页 tab

但它仍然偏“能用”，还不是“好用”：

1. 用户能看到记录，但很难快速找到某一次排盘
2. 卡片信息密度不统一，跨术数扫描成本高
3. 页面只有“筛选”和“刷新”，缺少搜索、排序、分组等检索能力
4. 页面缺少“我现在看的是哪一类记录”的反馈
5. 历史页承担的是跨系统入口，但当前视觉和信息层级还偏列表工具页

因此，这个页面需要从“记录列表”升级为“历史检索与回顾中心”。

---

## 2. 设计目标

历史页的核心任务不是展示所有记录，而是帮助用户完成以下 4 件事：

1. 快速找到某次排盘
2. 快速回忆当时问了什么
3. 快速判断这条记录值不值得点进去
4. 快速整理、筛选、清理自己的历史

设计上要优先服务这 4 个任务，而不是优先展示术数细节。

---

## 3. 当前实现约束

为了避免再写出脱离代码现实的方案，先明确当前运行时约束。

### 3.1 多术数记录

历史页展示的是统一的 `DivinationResult` 列表，而不是某个术数的专属历史。

这意味着页面层必须优先显示“跨系统共通信息”：

- 问事摘要
- 术数系统
- 起卦方式
- 起卦时间
- 结果摘要

### 3.2 UI factory 分发

当前历史页通过 `DivinationUIRegistry` 把卡片内容分发给各系统 UI factory。

这带来两个现实：

1. 好处：不同术数可以自定义自己的摘要形式
2. 问题：如果完全交给 factory，页面层就无法保证跨系统的扫描一致性

### 3.3 `chromeless` 模式是正式能力

历史页不是只在独立路由中使用，还被嵌入首页 tab。

所以设计文档必须同时覆盖两种上下文：

- 独立页面模式：有 `AntiqueScaffold`、`AntiqueAppBar`
- 嵌入模式：只渲染 body 内容，不重复创建页面外壳

---

## 4. 过去设计中的问题

历史页未来演进时，需要避免重走这几类弯路。

### 4.1 问题一：把历史页做成“纯列表”

如果页面只有一串卡片，用户只能靠滚动寻找目标记录，随着数据增长体验会迅速变差。

**结论**：

- 历史页必须具备搜索、筛选、分组三种能力

### 4.2 问题二：把卡片完全交给系统自己画

完全放权给各系统 UI factory，会导致：

- 同一页内信息层级不统一
- 用户扫描路径混乱
- 某些系统卡片信息过多，某些过少

**结论**：

- 页面层应定义统一的历史卡片信息骨架
- 系统层只负责补充系统特有摘要

### 4.3 问题三：把历史页当成详情页缩略版

历史页的职责是“找”和“回忆”，不是“深读”。

如果在列表首屏塞入过多术数细节，会直接伤害检索效率。

**结论**：

- 卡片只保留必要摘要
- 深度内容进入详情页

### 4.4 问题四：不区分 P0 和增强项

收藏、批量整理、导出、最近查看等能力都很有价值，但不是当前数据结构的基础能力。

**结论**：

- 本文档把元素分成 P0 必需、P1 推荐、P2 增强三层

---

## 5. 页面总体结构

历史页建议采用以下结构：

```text
┌──────────────────────────────┐
│ 历史记录                     │
│ 搜索框                       │
│ 筛选/排序工具条              │
│ 当前筛选状态条（可选）       │
├──────────────────────────────┤
│ 今天                         │
│ 记录卡片 A                   │
│ 记录卡片 B                   │
├──────────────────────────────┤
│ 近 7 天                      │
│ 记录卡片 C                   │
├──────────────────────────────┤
│ 更早                         │
│ 记录卡片 D                   │
└──────────────────────────────┘
```

页面从上到下的阅读顺序应该是：

1. 我现在在哪
2. 我要怎么找
3. 当前筛选了什么
4. 有哪些记录
5. 每条记录值不值得点进去

## 6. 页面应包含的元素

## 6.1 P0 必需元素

这些元素是历史页成为“可用产品页”的最低要求。

### 6.1.1 顶部标题区

包含：

- 页面标题：`历史记录`
- 右上角操作：
  - 筛选
  - 刷新

独立页面模式下使用 `AntiqueAppBar`。  
嵌入模式下标题应由宿主页面负责，历史页 body 不再重复显示顶层大标题。

### 6.1.2 搜索框

应放在列表上方，优先级高于筛选菜单。

建议支持搜索：

- 问事关键词
- 卦名 / 课体摘要
- 术数名称
- 起卦方式名称

如果当前实现阶段还不接数据库搜索，P0 可以先做前端本地过滤。

### 6.1.3 筛选工具条

P0 最低应包含：

- 术数系统筛选
- 排序方式

排序建议：

- 最新优先
- 最早优先

系统筛选建议：

- 全部
- 六爻
- 大六壬
- 梅花易数
- 小六壬

### 6.1.4 当前筛选状态条

当用户应用筛选或搜索时，需要有显式反馈。

建议包含：

- 当前系统筛选
- 当前关键字
- 当前结果数
- 清除筛选按钮

当前实现中已有“筛选: XXX (N 条)”雏形，后续应扩展为统一状态条。

### 6.1.5 分组标题

列表建议按时间分组，而不是直接平铺。

P0 推荐分组：

- 今天
- 近 7 天
- 更早

这样做的原因：

- 用户天然按时间回忆
- 比单纯日期排序更易扫读
- 可以在数据量变多后仍保持结构感

### 6.1.6 历史记录卡片

每张卡片至少应包含以下元素：

- 系统标识
  - 术数名称
  - 可选图标或色标
- 主标题
  - 优先显示“问事摘要”
  - 没有问事时显示结果摘要
- 时间信息
  - 公历时间
- 起卦方式
  - 时间卦 / 钱币卦 / 报数起课 / 电脑卦等
- 结果摘要
  - 六爻：本卦 → 变卦
  - 大六壬：课体 / 三传摘要
- 进入详情的点击行为
- 删除行为

### 6.1.7 页面状态

P0 必须完整覆盖：

- 加载中
- 加载失败
- 空历史
- 搜索无结果

这几个状态不能只靠文本兜底，必须有明确的视觉层级和操作入口。

---

## 6.2 P1 推荐元素

这些元素会明显提升体验，但不一定要在第一版全部落地。

### 6.2.1 顶部摘要区

位于搜索框之上或之下，显示轻量概览：

- 总记录数
- 当前筛选范围内记录数
- 最近一条记录时间

这个区域不要做成厚重 dashboard，只做一行或一块轻摘要即可。

### 6.2.2 术数 badge / 标签体系

每条记录应至少有一枚系统标签：

- 六爻
- 大六壬
- 梅花易数
- 小六壬

标签建议使用 `AntiqueTag`，颜色按系统区分，但必须保持 antique 总体调性。

### 6.2.3 快速摘要预览

卡片可增加一行轻预览：

- 问题预览
- 结果预览

注意：只允许一行，不要把卡片扩展成详情页。

### 6.2.4 空状态引导动作

空历史状态下建议提供：

- 文案：`暂无历史记录`
- 操作按钮：`去起卦`

搜索无结果时建议提供：

- 文案：`没有找到相关记录`
- 操作按钮：`清除搜索 / 查看全部`

---

## 6.3 P2 增强元素

这些元素价值高，但当前仓库未必已有数据承载能力。

### 6.3.1 收藏

建议元素：

- 卡片收藏图标
- “只看收藏”筛选

### 6.3.2 个人笔记状态

如果记录支持额外笔记，可显示：

- `有笔记`
- `已解读`
- `AI`

### 6.3.3 批量管理

包括：

- 批量删除
- 批量收藏
- 批量导出

### 6.3.4 最近查看 / 常用记录区

可在列表顶端增加：

- 最近查看
- 固定记录

但这属于二阶段能力，不应抢占基础搜索与筛选的优先级。

---

## 7. 历史卡片信息结构

历史页成败的关键，不在页面壳，而在卡片的信息层级。

推荐层级如下：

### 第一层：我当时问了什么

优先显示：

- 用户输入的问事摘要

没有问事时，回退显示：

- `record.getSummary()`

### 第二层：这是什么类型的记录

显示：

- 术数系统
- 起卦方式
- 起卦时间

建议排列形式：

`六爻 · 时间卦 · 2026-04-18 10:42`

### 第三层：算出了什么

只显示一行核心摘要：

- 六爻：`天雷无妄 → 天风姤`
- 大六壬：`涉害课 · 初传申 中传子 末传辰`

### 第四层：附加状态

可选显示：

- 有 AI 解读
- 有笔记
- 已收藏

这层是辅助判断，不应上升到主标题层。

---

## 8. 架构建议

为了让历史页既统一又支持多术数扩展，建议采用以下责任分配。

### 8.1 页面层负责统一骨架

页面层负责：

- 搜索
- 筛选
- 排序
- 分组
- 状态页
- 卡片外层布局
- 手势和操作入口

页面层允许依赖的系统最小输出面，应以
`docs/architecture/divination-system-interface.md`
中的输入输出契约为准，而不是临时读取某个系统实现细节。

### 8.2 系统层负责提供系统摘要

系统 UI factory 负责：

- 系统特有摘要文本
- 系统特有的小视觉提示

### 8.3 不建议继续把整张卡片完全外包给 factory

最佳长期方向不是：

- `buildHistoryCard(result)` 返回整张完整卡片

而是更接近：

- 页面层统一渲染卡片外壳
- 系统层提供摘要模型或局部内容

因为历史页首先是跨系统检索页，其次才是系统个性化展示位。

如果整卡完全外包，页面层会失去统一信息结构。

---

## 9. 视觉建议

历史页应保持 antique 体系下的克制感，不应做成“沉重卷轴感”。

建议如下：

- 页面根：`AntiqueScaffold`
- 分组标题：`AntiqueSectionTitle`
- 分组分隔：`AntiqueDivider`
- 卡片：`AntiqueCard`
- 系统与状态标签：`AntiqueTag`
- 空状态轻装饰：`AntiqueWatermark`

视觉原则：

- 历史页优先“清晰”和“检索效率”
- 装饰感弱于首页和结果页
- 信息密度高于首页，低于详情页

---

## 10. 分阶段落地建议

### Phase 1：当前页升级为合格的历史页

目标：

- 补搜索
- 补排序
- 补时间分组
- 统一筛选状态条
- 统一空状态 / 搜索空状态

### Phase 2：统一卡片骨架

目标：

- 把历史页卡片从“factory 完整托管”调整为“页面层统一骨架 + 系统摘要插槽”

### Phase 3：增强管理能力

目标：

- 收藏
- 笔记状态
- 批量管理

---

## 11. 验收标准

历史页改版完成后，应至少满足以下标准：

1. 用户可以通过搜索或筛选在 3 步内定位一条记录
2. 页面支持时间分组，而不是单纯平铺长列表
3. 每条卡片都具备统一的基础信息层级：
   - 问事/主标题
   - 系统 + 起卦方式 + 时间
   - 结果摘要
4. 页面完整覆盖加载、错误、空历史、搜索空结果四类状态
5. 独立页面模式与 `chromeless` 模式都成立
6. 历史页在 antique 体系下保持一致视觉，但不牺牲检索效率

---

## 12. 结论

历史记录页不是“把所有记录列出来”就结束了。

它应该是一个兼顾：

- 检索
- 回忆
- 判断
- 管理

的跨术数中心页。

最优方案不是继续堆更多术数细节，而是先把以下几个基础问题做扎实：

- 搜索
- 筛选
- 分组
- 统一卡片骨架
- 完整状态页

只有这几个基础打稳，历史页才会真正从“工具列表”升级为“用户资产入口”。
